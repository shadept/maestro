- FIX (flaky tests): DONE — both CHORE-entry WATCH flakes were REAL product bugs, not test artifacts. (1) TurnQueue per-session FIFO VIOLATED (reproduced 8/12 runs, three different tests, e.g. same-session t2 starting before t1): FUR-13's `boss.work()` encoding spawns localConcurrency=cap independent pollers; SKIP LOCKED lets poller B claim a session's second turn while poller A holds its first, then client-side localGroupConcurrency admission runs in fetch-response ARRIVAL order — whichever response lands first runs, the earlier turn gets restore()d and starts late. Fix: dispatch rebuilt as a single in-process dispatcher fiber over pg-boss's public fetch/complete/fail/touch — one fetcher kills the cross-fetcher race by construction; fetch({batchSize: freeSlots, groupConcurrency: 1}) excludes active-turn sessions and admits ≤1 job/session atomically INSIDE the fetch statement (no restore path needed — losers stay 'created'); batch re-sorted by created_on because UPDATE...RETURNING doesn't preserve the CTE's ORDER BY; heartbeats via boss.touch every 30s; NOTIFY dropped (dispatcher polls at the 500ms floor, which was already the latency floor for the transitions that matter). INVARIANT documented: pg-boss's in-statement group ranking picks by job id, so per-session FIFO leans on TaskRunIds being monotonic UUIDv7 (guaranteed since the uuidv7 CHORE). Multi-process would need the SKIP LOCKED fallback with DB-side admission — comment updated. (2) Git worktree config-lock race (reproduced 13/15): concurrent `git worktree add -b` on one bare repo both write upstream-tracking config (the mirror refspec makes refs/heads/* look remote-tracking, so autosetupmerge fires) and git fails fast on config.lock instead of waiting — will happen in production whenever two sessions provision on one project. Fix: new RepoLocks service ("maestro/git/RepoLocks", per-project mutex via sync Semaphore.makeUnsafe get-or-create) serializing ensureClone (also closes the double-clone race), fetch (rewrites refs/heads/*), provision, and remove; defaultBranch stays unlocked (read-only, and locked ops call it — RepoLocks is not reentrant). Tests unchanged except git.test.ts providing RepoLocks.layer; per-session FIFO assertions kept strict, concurrency kept in the git test. Proof: 10/10 consecutive green runs per file (was 4/12 and 2/15), full gates green.
